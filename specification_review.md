# Jリーグ試合結果予測アプリ 仕様検証およびレビュー結果

提供いただいた仕様および `preprocess.py` の解析を行いました。結果として、いくつか矛盾点やシステムを構築する上で詳細を詰める必要がある不明点が見つかりました。以下のポイントについてご確認ください。

## 1. データリーク（情報漏洩）の懸念と学習・予測の不一致（🚨最重要）

**懸念事項：**
`preprocess.py` を確認したところ、試合結果と各種チーム指標を結合する際、試合ごとの識別子である `shotkey` と `team_id` をキーにして結合を行っています（例: `left_on=['shotkey', 'home_team_id']`）。

**矛盾点：**
- **学習時のデータ:** `shotkey` で結合しているということは、その試合（過去）の「事後のパフォーマンスデータ（実績）」をモデルに学習させてしまっている可能性があります（これをデータリークと呼びます）。
- **予測時のデータ（手順3・4）:** これから行われる試合のため、試合結果に紐づく `shotkey` の実績データは存在しません。そのため、最新の平均・総合的なチームデータ（CSV）を使うことになります。
- **結論:** 「学習データ（試合事後の実績値）」と「予測データ（試合前の最新指標）」でデータの質が違ってしまうため、このままモデルを作っても本番の予測では全く当たらないモデルになってしまう可能性が高いです。

**確認点：**
👉 学習用のデータベース内の指標データ（goal_points等）は、「その試合が始まる前の時点でのチーム指標」を持つように考慮されたデータでしょうか？それとも「その試合で発生した実績」でしょうか？予測用と学習用で指標の意味合いがズレていないか確認が必要です。

## 2. 実績データの蓄積フロー（手順6）について

**懸念事項：**
手順6にて「試合結果を目的変数とし、予想に使用した最新チームデータを学習データとして蓄積し次回以降のモデル学習に使用します」とあります。

**不明点：**
手順1では「過去データを**データベースのテーブルから取得**して `processed_data.csv` を出力」していますが、手順6で新しく蓄積するデータの保存先はどこになりますか？
- **A案:** 新しい試合が終わるたびに、**データベース（Supabase等）**にその試合レコードとチーム指標インサートし、次回以降はまた手順1からやり直す運用。
- **B案:** データベースはそのままに、出力後の `processed_data.csv` に対して直接行を行追加（追記）していく運用。

将来の運用を考えると、A案（データベースを正として蓄積する）が整合性が保ちやすいですが、現在の構想はどちらに近いでしょうか？

## 3. team_mapping の役割

**懸念事項：**
仕様に記載のあった `team_mapping` は `preprocess.py` 内では使用されていませんでした。

**確認点：**
これは、DB学習用データの作成時点では不要であり、「Webアプリで利用者が対戦カード（画面上のチーム名プルダウンなど）を選ぶとき」や、「予想結果の画面表示」の際に ID ↔ チーム名 の変換として使用する（フロントエンド・Webアプリ側で参照する）という認識でよろしいでしょうか？

## 4. dataフォルダ配下のCSVフォーマット

**不明点：**
手順3における「dataフォルダ配下のCSVファイル（最新チームデータ）」は、`preprocess.py` で Supabase から取得している 各テーブル（`goal_points`, `deffence_points` など）と全く同じカラム構成・フォーマットを持っているという前提でよろしいでしょうか？（予測時に、学習データと全く同じ特徴量・同じカラム名を作り出す必要があるためです）
